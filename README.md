## 《云计算技术》期末实验大作业（分组）说明

### 1. 目标与原则

- **目标**：综合运用课程内容（云计算特征/服务模型、虚拟化、容器、OpenStack、K8s/云原生等）完成一个可复现、可验证的实验项目，并进行答辩。
- **三条硬性原则**：
  - **可复现**：别人的机器按你的文档能跑起来
  - **可验证**：有清晰的验证用例与结果证据
  - **可解释**：能讲清楚“为什么这么做”，而不仅是“我做了什么”

---

### 2. 分组与选题

- **建议分组人数**：4–5 人/组为主，允许 3 人/组（73 人建议约 15–20 组，按实际情况调整）。
- **选题（两种方式二选一）**：
  - **方式 A：从清单选题**：从 `选题列表.md` 中任选 1 题（清单仅提供方向与难度参考）。
  - **方式 B：自拟题（允许且鼓励）**：只要与云计算相关即可（IaaS/PaaS/SaaS、虚拟化、容器、K8s、云原生、可观测、DevOps、安全等），**先提交题目与范围给老师确认**，其余要求与评分标准不变。
- **同题说明**：同题可多组选，但实现细节需不同（规模/拓扑/镜像初始化/监控维度/自动化程度等）。

---

### 3. 仓库内容结构（提交时必须包含）

建议仓库目录如下（可按项目需要扩展）：

```text
.
├── README.md
├── 选题列表.md
├── 实验报告模板.md
├── report/
│   └── 《云计算技术》期末大作业报告.md
├── slides/
│   └── 答辩PPT.pdf
├── src/                # 如有代码/脚本/配置放这里
└── assets/             # 截图/拓扑图/日志片段等
```

---

### 4. 提交物清单

- **实验报告**：1 份，按 `实验报告模板.md` 撰写，提交到 `report/` 目录。
- **答辩 PPT**：1 份（建议导出 PDF），提交到 `slides/` 目录。
- **项目材料**（按选题）：脚本/配置文件/关键命令清单/代码等。
- **证据材料**：关键截图、日志输出、验证结果（可放 `assets/`）。

---

### 5. 命名规范（便于批改）

建议统一命名：

- **仓库名**：`cloud-final-班级-第X组-题目编号`  
  示例：`cloud-final-23计科1-第03组-A1`
- **分支名（强制，每组一个分支）**：`group/班级-第X组-题目编号`  
  示例：`group/23计科1-第03组-A1`
- **报告文件**：`《云计算技术》期末大作业报告.md`（或 pdf/docx，但推荐 md）
- **答辩文件**：`答辩PPT.pdf`

---

### 6. 报告与答辩要求

#### 6.1 报告硬性要求

- 必须包含：**环境与版本**、**实现步骤（含关键命令/配置解释）**、**验证用例与结果**、**排错记录（≥3）**、**分工与证据**。
- 截图可以有，但**不能只堆截图**：必须配文字解释。

#### 6.2 答辩建议结构（8–12 页）

- 项目背景与目标
- 关键概念与原理（对应课件）
- 架构/拓扑
- 关键实现与演示点
- 验证结果与数据/截图
- 排错与复盘
- 分工与贡献
- 总结与展望

#### 6.3 答辩时长（硬性要求）

- **每组答辩总时长不超过 10 分钟**（超时将被叫停）。
- **建议时间分配**：
  - 汇报/演示：6–7 分钟
  - 提问与追问：3–4 分钟
- **要求**：答辩内容必须覆盖“关键链路验证/结果证据/排错复盘/分工贡献”。

---

### 7. 评分细则（总分 100：代码/仓库 30 + 报告 20 + 答辩 50）

> 说明：自拟题与清单题一视同仁，按同一标准评分。若材料明显抄袭/买卖/伪造数据，按课程规定处理（通常为 0 分或取消答辩资格）。

#### 7.1 代码与仓库材料（30 分）

- **提交规范与证据（8 分）**
  - **分支提交**：每组必须在独立分支提交（符合第 8.0 规则）（4 分）
  - **提交记录**：有持续的 commit 历史，提交信息基本可读（2 分）
  - **材料齐全**：`report/`、`slides/`、`assets/`、`src/`（如适用）结构清晰（2 分）
- **可复现（10 分）**
  - **环境/版本/依赖明确**：别人按文档能搭建（5 分）
  - **一键/半自动复现**：脚本/Makefile/compose/yaml 等（满足其一即可）（5 分）
- **实现质量与工程化（8 分）**
  - **功能实现达标**：达到选题最低要求（或自拟题事先确认的最低要求）（4 分）
  - **配置规范**：关键配置文件、参数含义说明，避免硬编码敏感信息（2 分）
  - **安全与最小权限意识**：安全组/密钥/凭据处理合理（2 分）
- **验证与运维意识（4 分）**
  - **验证脚本/命令/用例**：能证明“确实做到了”（2 分）
  - **故障注入或真实问题处理**：能复现问题并定位解决（2 分）

#### 7.2 实验大作业报告（20 分）

- **结构完整性（4 分）**：按模板关键章节齐全（环境、步骤、验证、排错、分工、总结等）
- **原理与概念对应（4 分）**：能把课程概念（云计算特征/服务模型、虚拟化/容器、OpenStack/K8s 等）对应到项目设计与实现
- **实现过程可读可复现（6 分）**
  - 步骤清晰、关键命令/配置有解释（4 分）
  - 结果截图/日志不堆砌，能说明结论（2 分）
- **验证设计（3 分）**：验证用例（输入/预期/实际）充分，结论可信
- **排错复盘（3 分）**：≥3 条问题记录（现象-定位-根因-解决-预防）

#### 7.3 答辩（50 分）

- **PPT 结构与表达（15 分）**：逻辑清晰、重点突出、时间控制合理
- **现场演示与结果解释（20 分）**：演示关键链路（或回放关键证据），能解释“为什么这样设计/这样验证”
- **提问与追问（12 分）**：能回答原理、参数、排错思路、可用性/安全等问题
- **成员参与度（3 分）**：随机抽问时能体现分工与真实参与

#### 7.4 常见扣分项（直接扣）

- **未按分支提交**：-4 分（计入“提交规范与证据”）
- **缺少报告或答辩材料**：缺一项按该项大分扣完
- **无法复现且无有效证据**：相关部分酌情大幅扣分
- **答辩超时（>10 分钟）**：会被叫停，并影响“时间控制”相关得分

### 8. GitHub 提交方式（学生操作）

#### 8.0 分支提交规则（本课程要求）

- **每个小组必须在独立分支提交材料**：不要直接往 `main` 分支提交你们组的内容。
- **一个分支对应一个小组**：分支命名按第 5 节规范，例如 `group/23计科1-第03组-A1`。
- **评阅方式**：老师按分支查看各组提交历史与材料；如需汇总，可在截止后由老师统一合并或保留各组分支不合并。

#### 8.1 创建仓库并首次推送（示例命令）

1) 在 GitHub 网页端新建空仓库（不要勾选初始化 README）。

2) 本地执行（把 `<REMOTE_URL>` 换成你的仓库地址）：

```bash
git init
git add .
git commit -m "init: cloud computing final project"
git branch -M main
git remote add origin <REMOTE_URL>
git push -u origin main
```

#### 8.1.1 创建你们组的分支并在分支提交（必做）

把 `<BRANCH_NAME>` 换成你们组分支名（例如 `group/23计科1-第03组-A1`）：

```bash
git checkout -b <BRANCH_NAME>
git push -u origin <BRANCH_NAME>
```

之后你们组的所有材料更新都在该分支完成并推送：

```bash
git add .
git commit -m "update: report/slides/materials"
git push
```

#### 8.2 后续提交

```bash
git add .
git commit -m "update: report and materials"
git push
```
